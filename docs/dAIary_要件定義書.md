## 1. プロジェクト概要

### 1.1 プロダクトビジョン

dAIary（ダイアリー）は、写真で日々を記録し、AIが言葉を添えるフォトダイアリーアプリケーションである。生成AIを活用してハッシュタグや投稿文を自動生成し、誰でもおしゃれなSNS向けテキストを手軽に作成できる。生成されたテキストはクリップボードコピーやOS標準の共有機能を通じて任意のアプリへ連携可能。日々の写真記録から発信までをシームレスにつなぎ、個人ユーザーからビジネスユースまで幅広い層をターゲットとする。

本プロダクトは2つのエディションを提供する。

| エディション | ディレクトリ | 概要 |
|---|---|---|
| **オンライン版** | `apps/online/` | Supabase + Gemini APIを使用。認証・クラウド同期・課金機能あり |
| **オフライン版** | `apps/offline/` | SQLite + Gemma 4 E2Bを使用。完全オフライン動作。認証・課金・広告なし |

### 1.2 ターゲットユーザー

| ユーザー層 | 特徴 | 主な利用シーン |
|---|---|---|
| SNS発信者 | 投稿用テキストを効率的に作りたい | 日常的なSNS投稿文の作成補助 |
| 個人ユーザー | 写真を気軽に管理・共有したい | 旅行・イベントの写真整理 |
| 中小企業・店舗 | 商品やサービスのプロモーション素材を作りたい | 販促用キャプション・ハッシュタグの作成 |
| カメラ愛好家 | 撮影した写真を整理・管理したい | 作品のアルバム管理 |

### 1.3 対応プラットフォーム

- iOS 15.0以上
- Android 10（API Level 29）以上

> **注:** オンライン版・オフライン版ともに同一のプラットフォーム要件。ただしオフライン版のAI推論（MediaPipe LLM Inference API）は、Android 12（API Level 31）以上を推奨。

## 2. 機能要件（オンライン版）

> 以下セクション2はオンライン版（`apps/online/`）の機能要件である。オフライン版の機能要件はセクション2Aを参照。

### 2.1 カメラ撮影機能

#### F-CAM-001: カメラ撮影

- アプリ内からカメラを起動し写真を撮影できる
- フロントカメラ / リアカメラの切替に対応
- フラッシュのON / OFF / 自動の切替に対応
- グリッドライン（三分割構図ガイド）の表示切替が可能
- 撮影した写真はアプリ内ストレージおよび端末のフォトライブラリに保存

#### F-CAM-002: フォトライブラリからの取り込み

- 端末のフォトライブラリから既存の写真を選択・取り込みできる
- 複数枚同時選択に対応（最大10枚）

#### F-CAM-003: 基本的な写真編集

- トリミング（自由比率・1:1・4:3・16:9）
- 回転（90度単位）
- 明るさ・コントラスト・彩度の調整
- フィルタ適用（プリセット10種類以上）

### 2.2 生成AI機能

#### F-AI-001: ハッシュタグ自動生成

- 写真の内容をAIが解析し、関連するハッシュタグを自動生成する
- 用途に応じた最適なハッシュタグ数を提案
  - 少なめ（3〜5個）: X・Facebook向け
  - 多め（10〜30個）: Instagram向け
- 日本語 / 英語のハッシュタグに対応
- ユーザーが生成結果を編集・追加・削除できる
- 生成結果のクリップボードコピー機能
- トレンドハッシュタグの優先提案機能（将来拡張）

#### F-AI-002: 投稿文自動生成

- 写真の内容と指定された雰囲気に基づいて投稿文を自動生成する
- 選択可能な文章スタイル:
  - **ポエム風:** 感性豊かな表現、比喩を多用
  - **ビジネス風:** フォーマルで簡潔、プロフェッショナルな印象
  - **カジュアル風:** 親しみやすい口語体、絵文字を適度に使用
  - **ニュース風:** 事実ベースの客観的な文章
  - **ユーモア風:** ウィットに富んだ面白い表現
  - **カスタム:** ユーザーが自由にプロンプトで指示
- 文章長の指定（短文・中文・長文）
- 日本語 / 英語での生成に対応
- 生成結果の再生成（リトライ）機能
- ユーザーによる生成結果の手動編集
- 生成結果のクリップボードコピー機能

#### F-AI-003: 共有機能

- 生成したハッシュタグ・投稿文をOS標準の共有シートで外部アプリに連携
- 写真 + 生成テキストをまとめて共有可能
- クリップボードへの一括コピー（ハッシュタグ + 投稿文）

#### F-AI-004: AIモデル構成

- メインモデル: Gemini 3 Flash Preview（高速レスポンス用）
  - ハッシュタグ生成
  - 投稿文生成
- サブモデル: Gemma 3（軽量推論・オフライン対応検討用）
  - 将来的なオンデバイス推論の候補

### 2.3 アルバム機能

#### F-ALB-001: アルバム管理

- ユーザーが自由にアルバムを作成・編集・削除できる
- アルバム名・カバー写真の設定
- 写真のアルバムへの追加・移動・複製
- アルバム内写真の並べ替え（日付順・手動）
- アルバムの共有リンク生成（公開 / 非公開設定）

#### F-ALB-002: 写真管理

- 写真一覧表示（グリッドビュー / リストビュー）
- 写真の詳細表示（EXIF情報・撮影日時・位置情報）
- お気に入りマーク
- 写真の削除（ゴミ箱機能・30日後自動削除）
- 写真の検索（AI解析によるタグベース検索）

#### F-ALB-003: クラウド同期

- Supabase Storageを利用した写真のクラウドバックアップ
- デバイス間の同期（同一アカウントでの利用）
- ストレージ容量管理（無料プラン: 1GB / サブスクリプション: 50GB）

### 2.4 課金・サブスクリプション

#### F-PAY-001: 無料プラン（Free）

- カメラ撮影・写真取り込み: 無制限
- AI生成回数: 1日10回まで
- クラウドストレージ: 1GB
- 広告表示あり

#### F-PAY-002: プレミアムプラン（Premium）

- 月額: 480円（税込）/ 年額: 4,800円（税込）
- AI生成回数: 無制限
- クラウドストレージ: 50GB
- 広告非表示
- 優先サポート

#### F-PAY-003: 課金基盤

- iOS: App Store In-App Purchase
- Android: Google Play Billing
- サブスクリプション管理（購入・更新・解約・復元）
- 無料トライアル期間: 7日間
- レシート検証はサーバーサイドで実施

#### F-PAY-004: 広告掲載

- 広告プラットフォーム: Google AdMob
- 広告形式:
  - バナー広告: 画面下部に常時表示
  - インタースティシャル広告: AI生成完了後に表示（5回に1回）
  - リワード広告: AI生成回数の追加獲得（1回視聴で+3回）
- プレミアムユーザーは全広告非表示

### 2.5 ユーザー管理

#### F-USR-001: 認証

- メールアドレス + パスワードによるサインアップ / ログイン
- Googleアカウントでのソーシャルログイン
- Apple IDでのサインイン（iOS必須）
- パスワードリセット（メール認証）
- Supabase Authを利用

#### F-USR-002: プロフィール

- ユーザー名・アバター画像の設定
- 利用統計（AI生成回数・ストレージ使用量）の表示
- アカウント削除機能

#### F-USR-003: 設定

- 通知設定（プッシュ通知の ON / OFF）
- デフォルト生成スタイルの設定
- デフォルト生成言語の設定
- テーマ設定（ライト / ダーク / システム連動）
- プライバシーポリシー・利用規約の表示

## 2A. 機能要件（オフライン版）

> オフライン版（`apps/offline/`）固有の機能要件。認証・課金・広告・クラウド同期は対象外。

### 2A.1 カメラ撮影機能

オンライン版と同一の仕様（F-CAM-001、F-CAM-002）を継承。撮影後の写真はローカルファイルシステム＋SQLiteに保存。

#### FO-CAM-001: ローカル写真保存

- 撮影・取り込みした写真をアプリ専用ディレクトリ（`daiary/photos/`）に保存
- 保存時にサムネイル（300x300px, JPEG品質80%）を自動生成し`daiary/thumbnails/`に保存
- 保存時にEXIFメタデータ（GPS、撮影日時、カメラ設定等）を自動抽出しDBに保存
- 保存時に画像の幅・高さを自動取得しDBに保存
- SQLiteの`photos`テーブルにメタデータを記録

### 2A.2 生成AI機能（オンデバイス）

#### FO-AI-001: オンデバイスAIによるハッシュタグ自動生成

- デバイス上のGemma 4 E2B（MediaPipe LLM Inference API）で写真を解析し、ハッシュタグを自動生成
- Platform Channel（MethodChannel）経由でKotlin/SwiftのネイティブAPIを呼び出し
- 日本語 / 英語のハッシュタグに対応
- 生成回数の制限なし（無制限）
- ネットワーク接続不要
- 生成結果はSQLiteの`ai_generations`テーブルに記録

#### FO-AI-002: オンデバイスAIによる投稿文自動生成

- オンライン版と同一のスタイル選択（ポエム風/ビジネス風/カジュアル風/ニュース風/ユーモア風/カスタム）
- 文章長の指定（短文・中文・長文）
- 日本語 / 英語での生成に対応
- ネットワーク接続不要

#### FO-AI-003: AIモデル管理

- 初回起動時にオンボーディング画面を表示し、AIモデルのダウンロードを案内
- Wi-Fi接続チェック: ダウンロード開始前にWi-Fi接続を確認。モバイルデータ通信時は確認ダイアログ表示
- モデルダウンロード（約1GB）の進捗表示
- モデルの削除機能（Settings画面から操作）
- モデル未ダウンロード時はAI生成画面で案内バナーを表示し、生成ボタンを無効化

#### FO-AI-004: AIモデル構成

- メインモデル: Gemma 4 E2B（Edge 2Bパラメータ、4bit量子化）
- 推論エンジン: MediaPipe LLM Inference API（Android: Kotlin、iOS: Swift）
- Flutter連携: Platform Channel（MethodChannel `com.daiary.offline/gemma`）

### 2A.3 アルバム機能

#### FO-ALB-001: ローカルアルバム管理

- アルバムの作成・編集・削除
- 写真の追加: フルスクリーンのフォトピッカーUIで複数選択可能
- アルバムカバー写真のサムネイル表示
- 公開/共有機能なし（シングルユーザーのため）

#### FO-ALB-002: ローカル写真管理

- 写真一覧表示（グリッドビュー / リストビュー切替）
- 画像読み込み最適化（cacheWidth指定によるメモリ効率化、フェードインアニメーション）
- お気に入りマーク（楽観的更新）
- 写真のソフトデリート（ゴミ箱機能）
- ゴミ箱の自動クリーンアップ（ソフトデリートから30日経過した写真を起動時に自動完全削除）
- 写真の完全削除（ファイル＋DB）
- 写真の復元（ゴミ箱から）
- 複数選択による一括操作（お気に入り、削除）
- AIタグによる写真検索

### 2A.4 データ管理

#### FO-DAT-001: SQLiteデータベース

- ローカルSQLiteにデータ保存（4テーブル: photos, albums, album_photos, ai_generations）
- 外部キー制約によるデータ整合性の維持
- パフォーマンス向上のためのインデックス設定

#### FO-DAT-002: ストレージ管理

- Settings画面でストレージ使用量（写真＋サムネイル）をリアルタイム表示
- 全データ削除機能（SQLite全テーブルクリア＋写真/サムネイルファイル一括削除）

### 2A.5 設定

#### FO-SET-001: アプリ設定

- デフォルト生成スタイルの設定
- デフォルト生成言語の設定（日本語 / 英語）
- テーマ設定（ライト / ダーク / システム連動）— MaterialAppに動的反映
- 設定はSharedPreferencesに永続化

## 3. 非機能要件

### 3.1 パフォーマンス

**オンライン版:**

- アプリ起動時間: 3秒以内（コールドスタート）
- カメラ起動時間: 1秒以内
- AI生成レスポンス時間: 5秒以内（ハッシュタグ）/ 8秒以内（投稿文）
- 画像アップロード: 10MB以下の写真を10秒以内に完了（4G回線）
- オフライン時の基本操作（撮影・アルバム閲覧）に対応

**オフライン版:**

- アプリ起動時間: 3秒以内（コールドスタート、DB初期化＋ゴミ箱クリーンアップ含む）
- カメラ起動時間: 1秒以内
- AI生成レスポンス時間: デバイス性能に依存（目安: 10〜30秒）
- 写真保存: ローカルファイル書き込み＋サムネイル生成＋DB登録を3秒以内
- 写真一覧: cacheWidth指定によるメモリ効率化、フェードインアニメーション

### 3.2 セキュリティ

**オンライン版:**

- 通信の全てをHTTPS/TLSで暗号化
- サーバーサイドでのレシート検証による課金不正防止
- Supabase Row Level Security（RLS）によるデータアクセス制御
- 個人情報保護法・GDPR対応のデータ取り扱い

**オフライン版:**

- 全データがデバイスローカルに保存され、外部に送信されない（プライバシー重視）
- AIモデルダウンロード時のみネットワーク通信が発生
- SQLiteの外部キー制約によるデータ整合性の維持

### 3.3 信頼性・可用性

**オンライン版:**

- サーバーサイド可用性: 99.5%以上（月間）
- データバックアップ: 日次自動バックアップ
- 障害時のグレースフルデグラデーション（AI機能停止時も撮影・アルバム機能は利用可能）

**オフライン版:**

- ネットワーク障害の影響なし（完全オフライン動作）
- AIモデル未ダウンロード時もカメラ・アルバム機能は利用可能（グレースフルデグラデーション）
- データはデバイスストレージに依存（バックアップはユーザー責任）

### 3.4 拡張性

- ユーザー数10万人規模までのスケーラビリティを初期設計で考慮（オンライン版）
- APIバージョニングによる後方互換性の維持（オンライン版）
- オフライン版: SQLiteマイグレーション機構による将来のスキーマ変更対応
- 将来的な機能追加（動画対応・AI画像加工・SNS直接連携等）を見据えたアーキテクチャ

## 4. システムアーキテクチャ概要

### 4.1 リポジトリ構成

モノレポ構成を採用し、モバイル（オンライン/オフライン）・バックエンド・DB定義・ドキュメントを単一リポジトリで管理する。

```
daiary/                    # Melos モノレポルート
├── apps/
│   ├── online/           # Flutter アプリ（オンライン版）
│   └── offline/          # Flutter アプリ（オフライン版）
├── packages/
│   └── shared/           # 共有コード（core/, config/theme.dart, services/share_service.dart 等）
├── backend/              # FastAPI サーバー（オンライン版）
├── web/                  # Next.js Web版（オンライン版）
├── supabase/             # マイグレーション・RLSポリシー
├── docs/                 # 共有ドキュメント
├── melos.yaml            # Melos モノレポ設定
├── pubspec.yaml          # ルート pubspec（Melos ワークスペース）
└── docker-compose.yml    # ローカル開発環境
```

## 5. データモデル（主要エンティティ）

### 5.1 オンライン版（Supabase PostgreSQL）

### users（ユーザー）
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | PK, Supabase Auth連携 |
| email | VARCHAR | メールアドレス |
| username | VARCHAR | 表示名 |
| avatar_url | TEXT | アバター画像URL |
| plan | ENUM | free / premium |
| plan_expires_at | TIMESTAMP | プラン有効期限 |
| daily_ai_count | INT | 当日のAI生成使用回数 |
| daily_ai_reset_at | DATE | AI回数リセット日 |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

### photos（写真）
| カラム | 型 | 説明 |
|---|---|---|
| id | UUID | PK |
| user_id | UUID | FK → users |
| storage_path | TEXT | Supabase Storage パス |
| thumbnail_path | TEXT | サムネイルパス |

## 7. 制約事項・リスク

### 7.1 技術的制約

**オンライン版:**

- Gemini APIの利用料金がユーザー数増加に伴いスケールするため、コスト管理が重要
- 写真のクラウドストレージ容量がユーザー数に比例して増加するため、ストレージコストの監視が必要

**オフライン版:**

- Gemma 4 E2BのFlutter統合はMediaPipe LLM Inference APIの安定性に依存
- オンデバイスAI推論はデバイス性能（RAM・CPU/GPU）に依存し、低スペック端末では推論速度が低下する
- モデルサイズ（約1GB）によりアプリのストレージ占有量が大きい
- 小型モデル（2Bパラメータ）のためGemini APIと比較してAI生成品質が劣る可能性がある

### 7.2 ビジネスリスク

- App Store / Google Playの審査ポリシー（特に広告に関するガイドライン）への準拠（オンライン版）
- 生成AIによる不適切コンテンツ生成のリスク（コンテンツフィルタリングの実装が必要）
- Gemini APIの仕様変更・料金改定への対応（オンライン版）
- MediaPipe / Gemmaモデルの更新・互換性維持（オフライン版）
